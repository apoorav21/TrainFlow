01 Executive Summary

TrainFlow is a full-stack, production-grade iOS fitness coaching application that combines real-time HealthKit biometrics, an AI coach powered by OpenAI GPT-5, a personalised training plan system, and a native Apple Watch companion — all backed by a fully serverless AWS infrastructure deployed via Infrastructure-as-Code.

Unlike generic fitness trackers, TrainFlow acts as a true digital coach: it reads 20+ health metrics directly from Apple's HealthKit, understands the user's race goals and injury history, and generates multi-week structured training plans tailored specifically to each athlete. Users can then have a continuous conversation with the AI coach — asking questions, requesting plan adjustments, logging workouts verbally, and receiving weekly performance summaries — all in plain English.

03 System Architecture

High-Level Architecture Overview

Request Lifecycle

1
Authentication: iOS app calls Amplify.Auth — tokens stored securely in Keychain. Every API call attaches the Cognito ID token as Authorization: Bearer <token>.
2
API Gateway: REST API validates the JWT against the Cognito User Pool. On success it forwards the request to the Lambda with requestContext.authorizer.claims.sub — the Cognito user ID.
3
Lambda Handler: Each function calls trainflow.shared.auth.extract_user_id(event) to get the user ID, validates the request body, and performs DynamoDB operations.
4
DynamoDB: All reads/writes go through trainflow.shared.db which handles Decimal ↔ float conversion and standard response formatting.
5
Response: Lambda returns a standard JSON envelope; trainflow.shared.response builds the CORS-enabled HTTP response.

04 Authentication System

TrainFlow implements a comprehensive authentication system built on AWS Cognito and the Amplify iOS SDK. It supports three distinct authentication flows, a password reset flow, and auto-session restoration at app launch.

Authentication Flows

1. Email / Password (Standard SRP)

The default flow uses Cognito's Secure Remote Password protocol. On sign-up, users provide their name, email, and password. After registration, Cognito triggers the tf-profile-init Lambda via a post-confirmation hook to create the DynamoDB user record.

2. Passwordless Email OTP (Custom Auth)

A fully custom auth flow implemented via three Cognito Lambda triggers:

DefineAuthChallenge
Controls the state machine. On first call, issues CUSTOM_CHALLENGE. On correct answer, issues tokens.
CreateAuthChallenge
Generates a 6-digit OTP, stores it in privateChallengeParameters, and sends it via AWS SES email.
VerifyAuthChallenge
Compares the user-submitted code against the stored answer, setting answerCorrect accordingly.
# tf-auth-create-challenge Lambda
import boto3, random

def handler(event, context):
    otp = f"{random.randint(0, 999999):06d}"
    event["response"]["publicChallengeParameters"] = {"type": "OTP"}
    event["response"]["privateChallengeParameters"] = {"answer": otp}
    event["response"]["challengeMetadata"] = "TRAINFLOW_OTP"

    boto3.client("ses", region_name="ap-south-1").send_email(
        Source=os.environ["FROM_EMAIL"],
        Destination={"ToAddresses": [event["request"]["userAttributes"]["email"]]},
        Message={
            "Subject": {"Data": "Your TrainFlow sign-in code"},
            "Body": {"Text": {"Data": f"Your code is: {otp}\n\nExpires in 3 minutes."}},
        },
    )
    return event

3. Forgot Password (3-Step Verified Reset)

A structured three-step flow in the iOS login screen:

1
Email entry — user enters email, app calls Amplify.Auth.resetPassword(for: email), Cognito sends OTP via email.
2
OTP verification — user enters the 6-digit code. Code is stored locally; no API call yet.
3
New password + requirements checklist — user sets a new password with a live requirements view (length, uppercase, lowercase, digit). Calls confirmResetPassword then auto-signs in.

Session Management

At app launch, AuthService.configureAmplify() calls checkCurrentSession() which fetches the active session from Amplify's Keychain cache. Amplify automatically refreshes expired tokens using the Refresh Token. A background task subscribes to Amplify.Hub.publisher(for: .auth) to react to sign-in, sign-out, and session expiry events in real time.

Error Handling

All Amplify errors are mapped through friendlyMessage(_:) — converting cryptic AWS error codes into user-readable messages:

// Swift — AuthService.swift
private func friendlyMessage(_ raw: String) -> String {
    let s = raw.lowercased()
    if s.contains("incorrect username or password") { return "Wrong email or password." }
    if s.contains("limitexceededexception") { return "Too many attempts. Wait a few minutes." }
    if s.contains("codemismatchexception") { return "Incorrect code. Please try again." }
    if s.contains("expiredcodeexception") { return "Code expired. Request a new one." }
    if s.contains("network") { return "Connection issue. Check your internet." }
    return "Something went wrong. Please try again."
}

06 AI Coaching Engine

The AI coaching system is TrainFlow's most sophisticated component. It uses OpenAI GPT-5 with function calling to create an agentic coach that can read real data, modify training plans, and maintain a long-term conversation with the user.

AI Coach View (iOS)

The AICoachView has two states managed by a single @State var showChat: Bool:

  • Home (Briefing) state — shown when no conversation exists. Displays today's health briefing, plan status, and quick-prompt chips.
  • Chat state — full conversation interface with message bubbles, a persistent input bar, and typing indicator.

When the user taps a quick prompt chip or types a message, it posts to tf-chat-message via APIClient.shared.post("/chat/message", body: ...) and appends the AI response to the local message list.

The AI Coach View listens to NotificationCenter for .openAICoachWithMessage notifications — other screens (e.g., the Today workout card) can deep-link into the AI chat with a pre-filled message. This enables contextual coaching flows, e.g., "My knee hurts, should I still do today's long run?"

# chat_handler.py — simplified agentic loop
def handle_chat(user_id, user_message, context):
    history = load_chat_history(user_id)      # from DynamoDB

    # Auto-summarise if history is long
    if len(history) > 20:
        summary = summarise_history(history[:-10])   # GPT-5-mini
        history = [summary_record] + history[-10:]

    messages = build_messages(context, history, user_message)

    # Agentic loop — up to 5 tool-call rounds
    for _ in range(5):
        response = openai.chat.completions.create(
            model="gpt-5",
            messages=messages,
            tools=TOOLS,
        )
        message = response.choices[0].message

        if not message.tool_calls:
            break                               # Final answer ready

        # Execute all tool calls, append results
        for call in message.tool_calls:
            result = execute_tool(user_id, call.function.name,
                                  json.loads(call.function.arguments))
            messages.append({"role": "tool", "content": str(result), ...})

    save_to_chat_history(user_id, user_message, message.content)
    return message.content

Context Pre-loading

Before every AI call, context_builder.py pre-fetches a rich snapshot of the user's state so the AI can answer questions without burning tool-call rounds on basic data fetches:

  • User profile (goals, fitness level, injuries, race target)
  • Active training plan metadata and current week
  • Last 7 days of health metrics (HRV, resting HR, sleep score, steps)
  • Last 14 days of workouts (both logged and HealthKit-synced)

Conversation Memory & Summarisation

Every message (user + assistant) is stored in the tf-chat DynamoDB table. When the history exceeds 20 messages, the oldest 10+ messages are sent to GPT-5-mini for compression into a single summary record. This keeps the context window efficient while preserving long-term memory of the user's coaching journey.

System Prompt

The system prompt (built by prompts.py) injects the full user context snapshot and establishes the AI's coaching persona: a knowledgeable, supportive endurance coach that speaks plainly, gives specific actionable advice, and never makes up data it hasn't fetched via tools.

Workout Deduplication

When the same workout appears in both TrainFlow logs and HealthKit (e.g., user ran via Apple Workout app), the app intelligently deduplicates:

// DynamicTrainingView.swift — deduplication logic
func deduplicateActivity() {
    var remove = Set<String>()
    for t in trainFlowActivities {
        for h in healthKitActivities {
            let overlap = abs(t.date.timeIntervalSince(h.date)) < 3600
            guard overlap, typeMatches(t.type, h.type) else { continue }
            // Prefer TrainFlow record if it has effortRating (user logged detail)
            if t.effortRating != nil { remove.insert(h.id) }
            else                    { remove.insert(t.id) }
        }
    }
    activities = (trainFlowActivities + healthKitActivities)
        .filter { !remove.contains($0.id) }
}

09 Workout Tracking

TrainFlow tracks workouts through two complementary channels: manual logging through the app UI, and automatic syncing from HealthKit/Apple Workout app. Both sources are unified in the activity feed with intelligent deduplication.

Manual Workout Logging

The WorkoutLogView modal allows users to record:

  • Actual distance completed (km)
  • Actual duration (minutes)
  • Effort rating on a 1–10 scale (subjective RPE)
  • Free-text notes ("felt strong", "knee ached at km 7")
  • Association to a planned workout day

The tf-workout-log Lambda stores this to the tf-workouts table with sort key {ISO-timestamp}#{UUID} and also calls tf-plan-update-day to mark the planned day as completed.

HealthKit Workout Sync

The tf-workout-healthkit-sync Lambda and its iOS counterpart in TrainingService.swift automatically sync workouts recorded by Apple Workout, Strava (if shared to Health), or any HealthKit-compatible app. Synced workouts carry:

  • Workout type (running, cycling, swimming, HIIT, strength training, etc.)
  • Distance in km
  • Duration in minutes
  • Active calories burned
  • Source app name
  • A source: "healthkit" flag for display differentiation

Workout Report (AI-Generated)

The tf-workout-report Lambda generates an AI-powered analysis of a completed workout. Given the workout data and the user's profile/plan, GPT-5 produces a brief coaching report: pacing analysis, effort assessment, recovery recommendations, and whether the session met the training stimulus goal.

Workout History Screen

The activity feed in DynamicTrainingView shows a unified, deduplicated list of recent workouts. Each entry shows:

  • Workout type icon and colour-coded label
  • Date, duration, distance
  • Effort rating (if logged via TrainFlow)
  • Source badge (TrainFlow vs HealthKit)
  • Tap to expand notes and see AI report

// ProgressView.swift — streak computation
private var currentStreak: Int {
    let sorted = trainingVM.workoutDays
        .filter { $0.isCompleted }
        .compactMap { DateFormatter().date(from: $0.scheduledDate) }
        .sorted(by: >)
    var streak = 0
    var check = Calendar.current.startOfDay(for: Date())
    for d in sorted {
        if Calendar.current.isDate(d, inSameDayAs: check) ||
           Calendar.current.isDate(d, inSameDayAs: check.dayBefore) {
            streak += 1
            check = Calendar.current.startOfDay(for: d)
        } else { break }
    }
    return streak
}

13 Database Design

TrainFlow uses a multi-table DynamoDB design with six purpose-built tables. All tables use userId as the partition key, ensuring all data for a user can be accessed efficiently. Tables that store multiple records per user use a composite sort key.

15 Security Architecture

Authentication Security

  • JWT Tokens — Cognito ID tokens (1-hour expiry) auto-refreshed by Amplify SDK using Refresh Tokens stored securely in iOS Keychain
  • API Gateway Authoriser — every API route requires a valid JWT; requests without a valid token receive HTTP 401
  • User ID isolation — all backend handlers extract the user ID from Cognito claims, never from the request body. A user cannot access another user's data even by modifying the request payload.
  • SRP (Secure Remote Password) — passwords never transmitted in plaintext; Cognito handles the SRP authentication protocol

Secret Management

  • OpenAI API Key — stored in AWS Secrets Manager, fetched at Lambda cold start. Never in source code or environment variables committed to git.
  • Amplify config — amplifyconfiguration.json is in .gitignore; contains non-secret Cognito pool IDs, not credentials
  • No plaintext secrets in repo — comprehensive .gitignore excludes .env, .env.*, amplifyconfiguration.json, awsconfiguration.json

IAM Least Privilege

  • Each Lambda has a dedicated execution role. DynamoDB policy grants only the 6 specific table ARNs (not wildcard *).
  • Secrets Manager policy scoped to arn:aws:secretsmanager:*:*:secret:trainflow/openai-api-key*
  • SES policy granted only to the tf-auth-create-challenge Lambda, not shared across all functions
  • tf-plan-generate can only be invoked by tf-chat-message via an explicit grantInvoke call

Data Privacy

  • HealthKit data is fetched on-device and sent to the backend only when the user has explicitly granted HealthKit permissions
  • The DELETE /account endpoint permanently deletes all data across all 6 DynamoDB tables for the user
  • CloudWatch log groups have 7-day retention and DESTROY removal policy to limit data persistence

Usability in Practice

TrainFlow's usability edge comes from designing every interaction around a real training workflow — not around what's easiest to build.

Frictionless First Run

New users land in the AI Coach tab and immediately start a natural conversation. There's no manual form-filling, no wizard with 12 steps, no drop-down menus for "training philosophy". The AI extracts everything it needs — race goal, fitness level, schedule, injuries — through questions that feel like talking to a real coach. Profile fields are saved automatically in the background as the user answers.

Everything in Context

The AI coach always has the user's current context — today's workout, last week's HRV trend, recent training load, injury notes — injected into every conversation. The user never needs to re-explain who they are. When they say "I'm tired today", the AI already knows their resting heart rate is up and their HRV dipped overnight.

Watch Integration That Actually Works

The Watch app shows today's exact workout prescription — not a generic countdown timer — because the phone pushes the day's data to the watch proactively on app foreground and whenever the Watch requests it. Workout phases (Warm Up → Interval 1 → Rest → Interval 2 → Cool Down) guide the athlete through each segment in real time. At the end, the effort rating is logged directly on the wrist before the user even reaches for their phone.

Plan Changes Feel Instantaneous

When a user asks the AI to swap tomorrow's tempo run for an easy recovery jog, the adapt_training_plan tool call updates DynamoDB in the same response turn. By the time the user closes the chat and opens the Training tab, the change is already there. There is no "syncing", no "refresh" button, no delay.

Health Data That Explains Itself

Raw numbers like "HRV: 42ms" mean nothing to most athletes. The Health tab's AI summary converts these into plain coaching language: "Your recovery metrics are trending down — 3 consecutive nights of below-average deep sleep combined with a 5-day training block suggests you're accumulating fatigue." The AI connects dots the user wouldn't see on their own.

Consistent Visual Language

Workout type colours — interval red, tempo yellow, long run orange, easy green, rest cyan — are applied identically in the training week grid, the workout detail sheet, the activity feed, and the Apple Watch screens. A user who sees an orange cell in their calendar grid instantly recognises it as a long run before reading any text.
